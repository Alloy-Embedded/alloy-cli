# Ship Quickstart, Examples, and Error Cookbook

## Why

VISION.md → Phase 6 promises *"5-min quickstart, error cookbook,
progressive examples"* as a non-negotiable DX win.  Today the
repo has reference docs (ARCHITECTURE, AI_INTEGRATION,
PROJECT_FORMAT, …) and the README — but no walkthrough that
takes a brand-new user from `pip install` to a blinking LED.

A first-time user who lands on the README needs:

1. A 5-minute path that produces something visible.
2. A handful of progressive examples (UART echo → SPI flash →
   DMA double-buffer) so they see how to grow the project.
3. An error cookbook so the first `alloy build` failure
   doesn't strand them in a search engine.

## What Changes

### `docs/QUICKSTART.md`

- Single-page walkthrough: install via uv / pip → `alloy new
  blinky --board nucleo_g071rb` → `alloy build` → `alloy
  flash`.
- Annotated screenshot of each step (reuses the existing
  `docs/images/` SVGs).
- Every command block carries the expected output so users can
  pattern-match.

### `docs/EXAMPLES/`

- `01-blinky/` — minimal GPIO toggle, ~30 lines.
- `02-uart-echo/` — UART loopback with the typed `add_uart`
  flow.
- `03-spi-flash/` — SPI driver against a synthetic `at25` part.
- `04-dma-double-buffer/` — DMA-driven UART RX, demonstrates
  `suggest_dma_pair`.

Each example ships:
- A `README.md` walking through the diff against the previous
  step.
- A complete `alloy.toml` so `alloy new --from-example
  <name>` (later proposal) can scaffold.

### `docs/ERROR_COOKBOOK.md`

- One section per `error_type` value emitted by
  `AlloyCliError`.
- Each section carries: trigger → message → fix →
  related-MCP-tool.
- Cross-linked from CLI / TUI error messages via "see
  ERROR_COOKBOOK#error-type".

### `docs/CHEATSHEET.md`

- Single-page CLI quick reference: every `alloy <cmd>` with
  a one-line description, common flags, and a tiny example.
- Generated by a small script that introspects the Click
  command tree so it can't drift.

### Scaffold helper

- `alloy new --from-example <name>` consumes the example's
  `alloy.toml` so the docs paths and the CLI agree.

## Impact

- README → QUICKSTART → first working firmware in five minutes
  on a known-good board.
- The example tree pins the typed kinds we shipped in
  `enrich-peripheral-kinds` so users see them in the wild.
- Errors stop being mysterious — the cookbook resolves the
  most common failures with a copy-pasteable command.

## What this DOES NOT do

- Does not introduce a multi-board "kitchen sink" example;
  each example targets one board.
- Does not auto-generate the cookbook from code; the prose
  needs human-curated context.
- Does not ship a `mkdocs` / `sphinx` site — Markdown stays
  the canonical format.  A static site can layer on later.
