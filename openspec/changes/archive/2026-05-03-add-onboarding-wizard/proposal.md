## Why

Wave 2 (`add-toolchain-installer`, archived once merged) gave alloy-cli
a working binary installer: ``alloy toolchain install --for <family>``
fetches every required tool into a content-addressed store and pins
the project's lockfile.  But the user has to *know* to run that
command after ``alloy new``, and they have to *know* that
``alloy doctor --fix`` won't actually install anything yet.  Two
correct knowledge requirements are one too many for a "five minutes
to first ELF" onboarding promise.

Wave 3 closes the loop.  ``alloy new`` learns to offer the install
right after scaffolding, ``alloy doctor --fix`` learns to install
missing non-vendor tools, a new ``alloy setup`` verb provides a
guided "fresh machine" wizard, and the TUI Onboarding screen finally
does what its placeholder promised.  All four flows route through a
shared orchestrator so the install logic lives in exactly one place.

## What Changes

- New core module ``alloy_cli.core.toolchain_orchestrator`` — single
  source of truth for "walk a family manifest, dispatch every
  non-vendor tool through ``tool_sources.adapter_for(...)`` +
  ``toolchain_manager.install(...)``, update the project lockfile,
  return per-tool ``InstallOutcome``."  Pure: no Click / Rich /
  Textual / sys.stdin coupling.  Each entry point owns its own UI;
  the orchestrator stays UI-free so tests can drive it without a TTY.
- ``alloy new`` gains ``--install-toolchain`` /
  ``--no-install-toolchain`` /  ``--auto`` flags + an interactive
  prompt.  Default behaviour: in a TTY, prompt with default Y; in CI
  / non-TTY, skip the install unless ``--install-toolchain`` is
  explicit.  The post-scaffold flow always prints the next-step
  command, even when the user declined.
- ``alloy doctor --fix`` adds a new auto-fixer category that walks
  the project's resolved family, identifies missing non-vendor
  tools, and dispatches each through the orchestrator.  Vendor tools
  STAY info-severity with the install_doc URL; never auto-fixed.
  Failures from one tool do NOT abort the rest — successive installs
  are independent.  Every outcome lands in the existing
  ``_print_fix_summary`` table.
- New ``alloy setup`` standalone command — guided interactive
  onboarding for a fresh machine.  Detects project state, shows
  installed/missing/vendor surface, offers install.  When run
  outside a project, embeds the ``alloy new`` flow first
  (``--board <id>`` prompted from a curated list).  Always ends
  with concrete "next steps" pointing at ``alloy build`` +
  ``alloy flash``.
- TUI ``OnboardingScreen`` replaces Wave 1's placeholder with the
  real wizard: family picker → install plan → live per-tool
  progress → "all set" CTA.  Reusable from ``alloy new`` interactive
  mode, ``alloy ui`` palette entry, and ``alloy setup`` when STDIN
  is a TTY and the user prefers a TUI over the line-based prompt.
- New MCP tool ``alloy.toolchain_apply_install_plan(family_id)`` —
  the mutating complement to Wave 2's ``toolchain_install_plan``.
  Takes a ``family_id``, walks the orchestrator, returns per-tool
  outcomes (``[{tool, version, sha256, skipped, reason,
  bytes_downloaded, install_doc_url?}]``).  Vendor tools surface
  as ``skipped=true, reason="vendor"`` with their per-OS URL so an
  LLM agent can show the user where to download manually — but the
  tool itself never spawns a download for them.  Errors propagate
  via the typed envelope (every Wave-2
  ``family-toolchain-installer-{checksum,download,extract,locked,
  store-corrupt}`` flows up unchanged).
- Documentation: ``docs/QUICKSTART.md`` rewritten to use the new
  flow ("two minutes to first ELF").  New
  ``docs/TOOLCHAIN_ONBOARDING.md`` covers when to reach for which of
  the four entry points.  ``docs/CHEATSHEET.md`` regenerated by
  ``scripts/generate_cheatsheet.py``.
- One new typed error ``OnboardingCancelledError`` (subclass of
  ``AlloyCliError``, ``error_type="onboarding-cancelled"``) so a
  user pressing Ctrl-C mid-prompt surfaces in audit logs as a
  recognisable code rather than a bare ``KeyboardInterrupt``.

## Capabilities

### New Capabilities

- ``toolchain-onboarding``: shared install orchestrator API, the
  four user entry points (``alloy new`` prompt, ``alloy doctor
  --fix`` extension, ``alloy setup`` verb, TUI Onboarding screen),
  the MCP apply tool, and the contract tests pinning a
  TTY-independent core walk.

### Modified Capabilities

- ``cli-surface``: ``alloy new`` gains the install-toolchain flags
  + post-scaffold prompt; ``alloy doctor --fix`` gains the
  toolchain auto-fixer; ``alloy setup`` lands as a new top-level
  verb.
- ``mcp-surface``: a new mutating tool
  ``alloy.toolchain_apply_install_plan`` joins the existing read
  + write surface; documents the two-phase preview→apply pattern
  with ``toolchain_install_plan``.
- ``tui-experience``: ``OnboardingScreen`` is promoted from a
  placeholder to a full wizard with family picker + plan + live
  progress + completion CTA.
- ``developer-experience``: ``docs/QUICKSTART.md`` rewritten to
  use the install-toolchain flow; new
  ``docs/TOOLCHAIN_ONBOARDING.md`` contributor reference.

## Impact

- **New code**:
    * ``src/alloy_cli/core/toolchain_orchestrator.py`` — shared
      install walker (~250 LOC including tests scaffolding).
    * ``src/alloy_cli/commands/setup.py`` — new ``alloy setup`` verb
      (~250 LOC).
    * ``docs/TOOLCHAIN_ONBOARDING.md`` — contributor + user-facing
      reference.
    * Substantial extension of ``src/alloy_cli/tui/screens/onboarding.py``
      (Wave-1 placeholder).
- **Modified code**:
    * ``src/alloy_cli/commands/new.py`` — flags + prompt + dispatch
      to the orchestrator.
    * ``src/alloy_cli/core/diagnose.py`` — new toolchain auto-fixer
      registered via ``AUTO_FIXERS``.
    * ``src/alloy_cli/commands/doctor.py`` — render the new fix
      outcomes (size + per-tool detail).
    * ``src/alloy_cli/main.py`` — register ``setup_command``.
    * ``src/alloy_cli/mcp/tools.py`` — register
      ``toolchain_apply_install_plan``.
    * ``src/alloy_cli/core/errors.py`` — add
      ``OnboardingCancelledError``.
    * ``src/alloy_cli/commands/toolchain.py`` — refactor
      ``install_command`` to dispatch through the new orchestrator
      (no behavioural change).
    * ``docs/QUICKSTART.md``, ``docs/CHEATSHEET.md``,
      ``docs/ERROR_COOKBOOK.md`` — content updates.
- **Dependencies**: no new runtime deps.  Stdlib (``input()``,
  ``sys.stdin.isatty()``) drives the line-based prompts; Textual
  (already pinned) drives the TUI screen.
- **Backward compatibility**: fully additive.  ``alloy new`` keeps
  exiting after scaffolding when STDIN is non-TTY; CI scripts that
  pipe ``alloy new`` through stdin keep working.  ``alloy doctor``
  without ``--fix`` is byte-identical to today.  Wave-2's
  ``alloy toolchain install`` keeps working even though its
  internal walk now goes through the orchestrator (orchestrator
  emits the same outcomes).
- **Out of scope (Wave 4)**: ``alloy reset / erase / monitor`` and
  EULA-gated guided detection.  Wave 4 also lands a
  ``WizardSession`` typed dataclass for resumable interactive
  flows; Wave 3's wizard keeps state purely in process memory.
